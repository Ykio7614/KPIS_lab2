# Архитектура системы учета объектов ИБ

## 1. Технологический стек

- Язык: `Python 3`
- GUI: `PyQt6`
- Графики и диаграммы: `matplotlib`
- Экспорт таблиц в PDF: `reportlab`
- Хранилище: `CSV-файлы`
- Сборка в исполняемый файл: `PyInstaller`

## 2. Логическая архитектура

Приложение разделено на три слоя:

1. `UI`
   Отвечает за формы, таблицы, кнопки, вкладки и пользовательские сценарии.

2. `Business / Application Logic`
   Координирует CRUD-операции, подготовку данных для поиска, экспорта и визуализации.

3. `Data Access`
   Инкапсулирует чтение и запись CSV, формирует “плоское” представление объекта для таблиц и графиков.

## 3. Модули приложения

### 3.1. Модуль ввода

Назначение:

- создание нового объекта учета;
- добавление произвольного набора параметров;
- автоподсказки по ранее введенным названиям и значениям;
- сохранение в CSV.

Основные элементы:

- поле `Объект / подразделение`;
- динамический список строк `Название параметра + Значение`;
- кнопки `Добавить параметр`, `Очистить форму`, `Сохранить объект`.

### 3.2. Модуль поиска

Назначение:

- фильтрация объектов по нескольким условиям;
- логика поиска `AND`;
- поддержка диапазона дат;
- вывод результатов в таблицу;
- редактирование;
- удаление объекта или отдельного параметра;
- экспорт результатов;
- просмотр журнала изменений.

Основные элементы:

- динамический набор фильтров `Поле + Значение`;
- для полей-даты появляются поля `с` и `по`;
- таблица результатов с прокруткой;
- кнопки `Показать`, `Редактировать`, `Удалить`, `Сброс поиска`;
- кнопки выгрузки `DOC`, `CSV`, `PDF`;
- кнопка просмотра журнала изменений.

### 3.3. Модуль визуализации

Назначение:

- чтение агрегированных данных из CSV;
- выбор типа диаграммы;
- выбор осей `X` и `Y`;
- построение диаграммы;
- экспорт изображения.

Поддерживаемые виды:

- столбчатая диаграмма;
- линейный график;
- круговая диаграмма.

Экспорт:

- `JPG`
- `JPEG`
- `PNG`
- `PDF`

### 3.4. Хранилище данных

Назначение:

- хранение объектов;
- хранение параметров;
- хранение истории изменений;
- обеспечение целостности данных при CRUD-операциях.

## 4. Структура CSV-файлов

### `data/objects.csv`

Хранит объект верхнего уровня.

| Поле | Назначение |
|---|---|
| `object_id` | Уникальный идентификатор объекта |
| `object_name` | Название объекта / подразделения |
| `created_at` | Дата и время создания |
| `updated_at` | Дата и время последнего изменения |

### `data/parameters.csv`

Хранит параметры объекта в нормализованном виде: один параметр = одна строка.

| Поле | Назначение |
|---|---|
| `parameter_id` | Уникальный идентификатор параметра |
| `object_id` | Внешний ключ на `objects.object_id` |
| `parameter_name` | Название параметра |
| `parameter_value` | Значение параметра |

### `data/change_log.csv`

Хранит историю изменений.

| Поле | Назначение |
|---|---|
| `log_id` | Уникальный идентификатор записи лога |
| `object_id` | Идентификатор объекта |
| `object_name` | Имя объекта на момент изменения |
| `field_name` | Измененное поле |
| `old_value` | Старое значение |
| `new_value` | Новое значение |
| `action` | Тип действия: `create`, `update`, `delete`, `delete_object`, `delete_parameter` |
| `changed_at` | Дата и время изменения |

## 5. Почему выбрана именно такая схема

Такое разбиение позволяет:

- хранить объект и параметры независимо;
- добавлять новые параметры без изменения структуры `objects.csv`;
- строить выборки и фильтры по произвольным полям;
- сохранять журнал изменений отдельно от текущего состояния;
- выполнять каскадное удаление параметров при удалении объекта.

## 6. Поток данных

1. Пользователь создает объект в модуле ввода.
2. UI передает данные в `CsvRepository`.
3. `CsvRepository` записывает объект в `objects.csv`, параметры в `parameters.csv`, а изменения в `change_log.csv`.
4. Модуль поиска читает те же CSV и формирует “плоскую” таблицу вида `Объект + все параметры`.
5. Модуль визуализации использует это же представление для построения агрегированных графиков.

## 8. Принципы расширения

Система легко расширяется:

- можно заменить CSV на SQLite/PostgreSQL без полной переделки UI;
- можно добавить новые форматы экспорта в `services/export_service.py`;
- можно вынести репозиторий в отдельный слой сервисов и покрыть тестами;
- можно добавить справочники параметров, авторизацию и роли пользователей.
